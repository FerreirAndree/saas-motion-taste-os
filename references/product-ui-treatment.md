# Product UI Treatment

## Principle

Show software as a designed object, not a flat screen recording. The interface should feel real enough to trust and composed enough to admire.

## UI Fidelity Levels

Choose a fidelity level deliberately:

- **Literal UI**: Actual product screen with minimal styling changes. Use for credibility, demos, and proof.
- **Cleaned UI**: Real UI simplified for legibility, spacing, and focus. Use for most launch films.
- **Abstracted UI**: Product logic represented as cards, panels, charts, nodes, or flows. Use for explaining concepts.
- **Symbolic UI**: The product becomes an icon, object, or spatial metaphor. Use for hooks, transitions, and brand moments.

Move between levels intentionally. Do not stay abstract so long that the product feels imaginary.

## Legibility Rules

- Enlarge the relevant feature beyond real UI scale when needed.
- Remove irrelevant navigation, table columns, empty states, and chrome.
- Use real labels where they prove the product.
- Blur, dim, or crop nonessential regions.
- Keep text readable during holds, not during fast motion.
- Avoid tiny mock data that looks decorative.

## UI as Cinema

Treat UI surfaces like cinematic props:

- Give key panels mass through shadow, edge light, or depth.
- Use parallax to separate foreground action from background context.
- Let highlights travel through the interface to guide reading.
- Make state changes visible: selected, running, completed, approved, synced.
- Use cursor, command palette, keyboard input, or system activity only when it adds causality.

## Product Proof Patterns

### Input to Output

Use when proving automation or intelligence.

- Show the input clearly.
- Show the system processing in a product-specific way.
- Reveal the output with a clean hold.
- Show why the output is useful.

### Selection to Insight

Use for analytics and knowledge tools.

- Show a field, query, filter, or question.
- Animate narrowing, ranking, or grouping.
- Land on one insight, not a wall of charts.

### Command to Workflow

Use for productivity and agentic tools.

- Show a user command or intent.
- Show the workflow steps as connected states.
- Mark completion visibly.
- End with a status or artifact.

### Integration to System

Use for platforms.

- Show one integration connecting.
- Show data entering a shared model.
- Show a downstream surface updating.

## Mock Data Rules

Mock data should look plausible and strategically chosen:

- Use names, labels, and metrics that reinforce the product's value.
- Avoid lorem ipsum.
- Avoid meaningless charts.
- Avoid too many perfect numbers.
- Use realistic variability.
- Keep sensitive or regulated domains calm and credible.

## Screen Composition

- Favor one hero surface over many equal panels.
- Use supporting panels as context, not clutter.
- Crop close enough to read the feature.
- Pull back only when showing system breadth.
- Avoid endless floating cards unless their spatial arrangement explains something.

## Interaction Details

Use micro-interactions to create trust:

- hover state before selection
- button press with small depth change
- row selection with highlight and detail panel update
- status pill changing state
- chart tooltip revealing exact value
- command accepted and executed
- synced indicator or checkmark resolving quietly

Do not add cursor movement if the product is being positioned as autonomous unless the user's control is part of the message.
